# Development Workflow

이 문서는 사용자 요청을 받아 개발, 테스트 코드 작성, 테스트 실행까지 완료하기 위한 표준 흐름을 정의한다. 현재 프로젝트의 Express/Prisma/Jest 구조와 `.claude/agents`의 서브에이전트 구성을 기준으로 한다.

## 목표

- 개발 요청을 코드 변경과 검증 완료 상태까지 일관되게 처리한다.
- 구현, 테스트 작성, 테스트 실행의 책임을 분리한다.
- 실제 DB를 사용하는 테스트 환경에서 운영/개발 DB 오염을 방지한다.
- 최종 응답에는 변경 내용, 검증 결과, 남은 리스크를 명확히 남긴다.

## 참여 역할

| 역할 | 책임 |
| --- | --- |
| 사용자 | 개발 요구사항, 기대 동작, 제약 조건, 완료 기준 전달 |
| 메인 에이전트 | 요구사항 분석, 코드 변경, 영향 범위 파악, 서브에이전트 작업 연결, 최종 정리 |
| `unit-test-agent` | 변경 내용과 테스트 계획서를 바탕으로 테스트 코드 작성 또는 보강 |
| `integration-test-agent` | 빌드/테스트 실행, 실패 분석, 테스트 리포트 작성 |
| `docs-agent` | API 문서 작성 및 수정 |

## 기본 흐름

```text
사용자 개발 사항 전달
  -> 메인 에이전트가 요구사항과 코드베이스 분석
  -> 메인 에이전트가 구현
  -> unit-test-agent가 테스트 코드 작성/보강
  -> integration-test-agent가 테스트 실행
  -> [실패 시] 구현 결함이면 메인 에이전트가 코드 수정 후 테스트 재실행으로 복귀
               테스트 결함이면 unit-test-agent가 테스트 수정 후 재실행으로 복귀
  -> 문서 업데이트, 최종 요약 및 커밋 메시지 제안
```

## 1. 개발 사항 접수

사용자 요청에서 다음 항목을 우선 확인한다.

- 변경 목적
- 대상 API, 모듈, 문서
- 정상 동작 기준
- 실패/예외 케이스
- 기존 동작과 달라지는 점
- 테스트 완료 기준

불명확하지만 합리적으로 추정 가능한 항목은 메인 에이전트가 보수적으로 결정한다. 여러가지 해석이 가능하거나, 더 단순한 접근 방식이 있거나, 불분명한 점이 있다면 멈추고 질문하여 확인한다. 데이터 삭제, 스키마 변경, 인증/권한 정책 변경처럼 위험한 결정은 사용자 확인을 받는다.

## 2. 사전 분석

메인 에이전트는 구현 전에 관련 근거를 확인한다.

- `CLAUDE.md` — 아키텍처 요약, 명령어, 인증 방식 등 프로젝트 운영 규칙
- `docs/spec.md` — 도메인 모델, 비즈니스 규칙, API 개요
- `README.md`
- `docs/ARCHITECTURE.md`
- `docs/research.md`
- 관련 API 문서: `docs/*-api.md`
- 진입점: `index.ts`
- 라우터: `routes/index.ts`
- 관련 컨트롤러: `controllers/*.ts`
- 데이터 모델: `schema.prisma`
- 테스트 계획서: `test/testplan-*.md`
- 기존 테스트: `test/test-*.ts`

분석 결과로 영향 범위와 테스트 필요 항목을 정리한다.

## 3. 구현

메인 에이전트가 구현을 수행한다.

- 기존 구조, 스타일을 우선 따른다.
- 작은 변경은 컨트롤러/테스트 수준에서 최소 수정한다.
- 공통 동작이 반복되거나 위험이 크면 헬퍼 또는 미들웨어 도입을 검토한다.
- Prisma 스키마 변경이 있으면 마이그레이션, Prisma Client 재생성, 테스트 영향까지 함께 고려한다.
- `.env`, 운영 데이터, 시크릿 값은 수정하거나 노출하지 않는다.
- 요청받지 않은 기능은 추가하지 않는다.
- 더 적은 코드로 작성할 수 있다면 다시 작성한다.
- 시니어 엔지니어가 이 코드를 보고 과도하게 복잡하다고 말할 수준이라면 더 단순화한다.

구현 후 메인 에이전트는 테스트 작성 서브에이전트에 넘길 정보를 준비한다.

- 변경 파일
- 변경된 동작
- 추가해야 할 정상/실패 테스트
- DB나 세션 등 테스트 전제조건

## 4. 테스트 코드 작성

테스트 코드 작성은 `unit-test-agent`의 책임이다.

`unit-test-agent`는 다음을 따른다.

- `test/testplan-*.md`와 기존 테스트를 확인한다.
- 변경된 기능의 정상 케이스와 실패 케이스를 추가한다.
- 서비스 그룹별 테스트 파일을 사용한다.
  - 인증: `test/test-auth-api.ts`
  - 도서: `test/test-book-api.ts`
  - 독서 기록: `test/test-reading-log-api.ts`
  - 메모: `test/test-memo-api.ts`
  - 통계: `test/test-stats-api.ts`
- 기존 테스트 파일은 덮어쓰지 않고 필요한 케이스만 추가한다.
- 세션 유지가 필요한 테스트는 `request.agent(app)`을 사용한다.
- DB 사이드 이펙트 검증이 필요한 경우 Prisma 직접 조회를 사용할 수 있다.

테스트 코드 변경 후 최소한 TypeScript 빌드가 깨지지 않아야 한다.

## 5. 테스트 실행

테스트 실행은 `integration-test-agent`의 책임이다.

실행 전 확인:

- `.env.test` 존재
- `NODE_ENV=test`
- `DATABASE_URL`이 테스트 DB를 가리킴
- `test/helpers.ts`의 `cleanDb()` 보호 장치가 유지됨
- 테스트 DB 연결 가능

빌드 및 스키마 확인 (테스트 실행 전 순서대로 수행):

```bash
npx prisma validate
npm run build
```

변경 범위가 좁으면 관련 도메인 테스트를 먼저 실행한다.

```bash
npm run test:auth
npm run test:book
npm run test:reading-log
npm run test:memo
npm run test:stats
```

릴리스 전 또는 공통 코드 변경 후에는 전체 테스트를 실행한다.

```bash
npm test
```

테스트 실행 후 `integration-test-agent`는 통과/실패 수, 실패 원인, 권장 조치를 정리한다. 정리 내용은 `test/integration-report_YYYYMMDDHHmmss.md`에 작성한다. (예: `integration-report_20260616153045.md`)

## 6. 실패 처리

테스트가 실패하면 다음 순서로 분류한다.

| 분류 | 기준 | 담당 |
| --- | --- | --- |
| 구현 결함 | 기대 동작과 실제 코드 동작 불일치 | 메인 에이전트 |
| 테스트 결함 — 코드 오류 | assertion·픽스처·전제조건 등 테스트 코드 자체의 오류 | `unit-test-agent` |
| 테스트 결함 — 설계 판단 오류 | 테스트 범위·케이스 선정이 잘못된 경우 | 메인 에이전트 |
| 환경 문제 | DB 연결, 마이그레이션, 권한, 샌드박스 문제 | 메인 에이전트 |
| 데이터 충돌 | 테스트 DB 초기화 누락 또는 순서 문제 | 메인 에이전트 |

수정 후 관련 테스트를 재실행한다. 동일 문제가 반복되면 원인과 차단 조건을 사용자에게 명확히 보고한다.

## 7. 완료 기준

개발 건은 다음 조건을 만족하면 완료로 본다.

- 요청된 기능 또는 수정이 구현됨
- 관련 테스트 코드가 추가 또는 보강됨
- `npm run build` 통과
- 관련 도메인 테스트 통과
- 공통 영향이 있으면 `npm test` 통과
- 남은 리스크와 미확인 항목이 최종 응답에 명시됨
- 추천 커밋 메시지가 제공됨

## 8. 문서 업데이트

기능 변경이 있으면 다음 기준으로 관련 문서를 업데이트한다.

| 변경 유형 | 업데이트 대상 |
| --- | --- |
| API 엔드포인트 추가·수정·삭제 | `docs/*-api.md` 해당 도메인 문서 |
| 도메인 모델·비즈니스 규칙 변경 | `docs/spec.md` |
| 새로운 테스트 시나리오 추가 | `test/testplan-*.md` 해당 도메인 계획서 |
| 아키텍처·기술 스택 변경 | `docs/ARCHITECTURE.md` |
| 명령어·환경·운영 규칙 변경 | `CLAUDE.md` |

단순 버그 수정이나 내부 리팩터링처럼 외부 동작이 달라지지 않는 경우에는 문서 업데이트를 생략할 수 있다.

## 테스트 DB 안전 기준

이 프로젝트의 테스트는 실제 MySQL DB를 사용한다. 다음 정책을 반드시 지킨다.

- 테스트는 `.env.test`만 사용한다. 운영 DB 또는 개발 DB를 테스트 대상으로 사용하지 않는다.
- `cleanDb()`는 `NODE_ENV === "test"`이고 DB 이름에 `test`가 포함될 때만 실행되어야 한다.
- DB 관련 테스트 명령은 WSL/Windows MySQL 연결과 Codex 샌드박스 권한의 영향을 받을 수 있다.

## 서브에이전트 사용 기준

- API 문서 작성/수정이 주된 작업이면 `docs-agent`를 사용한다.
- 테스트 케이스 추가 또는 누락 보강이 필요하면 `unit-test-agent`를 사용한다.
- 테스트 실행과 결과 리포트가 필요하면 `integration-test-agent`를 사용한다.
- 단순 오타 수정이나 문서 소폭 수정처럼 테스트 영향이 없는 작업은 메인 에이전트가 직접 처리할 수 있다.

## 최종 응답 형식

작업 완료 시 메인 에이전트는 다음을 간결히 보고한다.

- 변경 요약
- 테스트/빌드 결과
- 실패 또는 미실행 항목
- 남은 리스크
- 추천 커밋 메시지

예시:

```text
변경: 도서 등록 검증 로직과 관련 테스트를 추가했습니다.
검증: npm run build 통과, npm run test:book 통과.
리스크: 전체 테스트는 실행하지 않았습니다.
커밋 메시지: fix: 도서 등록 입력 검증 추가
```
