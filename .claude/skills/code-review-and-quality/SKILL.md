# Code Review and Quality

## 개요

품질 게이트를 활용한 다차원 코드 리뷰. 모든 변경 사항은 병합 전에 검토되며 예외는 없습니다. 리뷰는 정확성, 가독성, 아키텍처, 보안 및 성능의 다섯 가지 측면을 다룹니다.

**승인 기준:** 코드의 전반적인 건전성을 확실히 개선하는 변경 사항이라면 완벽하지 않더라도 승인하세요. 완벽한 코드는 존재하지 않으며, 목표는 지속적인 개선입니다. 자신이 작성한 방식과 정확히 일치하지 않는다고 해서 변경을 막지 마세요. 코드베이스를 개선하고 프로젝트 규칙을 준수한다면 승인하십시오.

## 사용 시점

- PR이나 변경 사항을 병합하기 전에
- 기능 구현을 완료한 후
- 다른 에이전트나 모델이 평가해야 할 코드를 생성했을 때
- 기존 코드를 리팩토링할 때
- 버그 수정 후 (수정 내용과 회귀 테스트 결과를 모두 검토하십시오)

## Five-Axis 검토

모든 코드 리뷰는 다음 기준에 따라 코드를 평가합니다.

### 1. 정확성

해당 코드는 주장하는 기능을 제대로 수행합니까?

- 사양이나 작업 요구 사항과 일치합니까?
- 예외 상황(null, 빈 값, 경계값)은 처리됩니까?
- 오류 경로도 처리되나요 (정상 경로만 처리되는 것이 아니라)?
- 모든 테스트를 통과했나요? 테스트가 실제로 올바른 항목을 테스트하고 있나요?
- 오프셋 오류, 경쟁 조건 또는 상태 불일치가 있습니까?

### 2. 가독성 및 간결성

다른 엔지니어(또는 담당자)가 작성자의 설명 없이도 이 코드를 이해할 수 있을까요?

- 이름은 설명적이며 프로젝트 규칙과 일관성이 있습니까? (맥락 없이 `temp`, `data`, `result`와 같은 이름은 사용하지 마십시오.)
- 제어 흐름이 직관적인가요 (중첩된 삼항 연산자나 깊은 콜백 함수를 피해야 하나요)?
- 코드가 논리적으로 구성되어 있습니까(관련 코드가 그룹화되어 있고 모듈 경계가 명확합니까)?
- 간소화해야 할 "교묘한" 기술들이 있나요?
- **이 코드를 더 적은 줄로 작성할 수 있을까요?** (100줄이면 충분한데 1000줄이나 쓰는 건 문제가 있습니다.)
- **추상화는 그 복잡성에 걸맞은 가치를 지니고 있는가?** (세 번째 사용 사례가 나올 때까지 일반화하지 마십시오)
- 주석이 모호한 의도를 명확히 하는 데 도움이 될까요? (단, 명백한 코드에는 주석을 달지 마세요.)
- 사용되지 않는 코드 조각(예: 아무 기능도 하지 않는 변수(`_unused`), 하위 호환성을 위한 임시 코드, 또는 `// removed` 주석)이 있습니까?

### 3. 아키텍처

변경 사항이 시스템 설계에 부합합니까?

- 기존 패턴을 따르는가, 아니면 새로운 패턴을 도입하는가? 새로운 패턴이라면, 그 타당성은 있는가?
- 모듈 경계를 명확하게 유지합니까?
- 공유해야 할 코드 중복이 있습니까?
- 종속성들이 올바른 방향으로 흐르고 있습니까(순환 종속성이 없습니까)?
- 추상화 수준이 적절한가요 (과도하게 복잡하지 않고, 지나치게 결합되어 있지 않은가요)?

### 4. 보안

자세한 보안 지침은 `보안 및 강화`를 참조하세요. 변경 사항으로 인해 취약점이 발생합니까?

- 사용자 입력은 유효성 검사 및 검증을 거치나요?
- 기밀 정보는 코드, 로그, 버전 관리 시스템에서 철저히 보호되나요?
- 필요한 곳에서 인증/인가 확인이 이루어지고 있습니까?
- SQL 쿼리는 매개변수화되어 있습니까(문자열 연결은 사용하지 않습니까)?
- XSS 공격을 방지하기 위해 출력값이 암호화되어 있습니까?
- 종속성은 알려진 취약점이 없는 신뢰할 수 있는 출처에서 제공되었습니까?
- 외부 소스(API, 로그, 사용자 콘텐츠, 설정 파일)의 데이터는 신뢰할 수 없는 것으로 간주됩니까?
- 외부 데이터 흐름은 로직이나 렌더링에 사용하기 전에 시스템 경계에서 유효성 검사를 거치나요?

### 5. 성능

자세한 프로파일링 및 최적화는 `performance-optimization`를 참조하세요. 변경 사항으로 인해 성능 문제가 발생합니까?

- N+1 쿼리 패턴이 있나요?
- 무한 루프나 제약 없는 데이터 가져오기가 있습니까?
- 동기 작업 중 비동기 작업으로 변경해야 할 부분이 있나요?
- UI 구성 요소에서 불필요한 재렌더링이 있습니까?
- 목록 엔드포인트에 페이지네이션이 누락된 부분이 있나요?
- 핫 패스에 대형 객체가 생성되었습니까?

## 크기 변경

작고 집중적인 변경은 검토가 쉽고, 병합 속도가 빠르며, 배포가 더 안전합니다. 다음 크기를 목표로 하세요.

```
약 100줄 수정됨 → 양호. 한 번에 검토 가능.
약 300줄 변경됨 → 논리적인 변경 사항이 하나뿐이라면 허용 가능.
약 1000줄이 변경되었습니다 → 너무 큽니다. 분할하세요.
```

**"하나의 변경 사항"으로 간주되는 것:** 제출 후에도 시스템이 정상적으로 작동하고, 관련 테스트를 포함하며, 하나의 문제를 해결하는 독립적인 수정 사항입니다. 전체 기능이 아닌 기능의 일부를 의미합니다.

**변화 규모가 너무 클 때 분할 전략:**

|전략 | 방법 | 시기 |
|----------|-----|------|
| **스택** | 작은 변경 사항을 제출하고, 이를 기반으로 다음 변경 사항을 시작하세요 | 순차적 종속성 |
| **파일 그룹별** | 검토자가 다른 그룹별 변경 사항 분리 | 공통 관심사 |
| **수평적** | 먼저 공유 코드/스텁을 생성한 다음 소비자를 생성합니다 | 계층형 아키텍처 |
| **수직적** | 기능을 더 작은 풀스택 부분으로 분할 | 기능 작업 |

**대규모 변경이 허용되는 경우:** 검토자가 모든 줄을 검토할 필요 없이 의도만 확인하면 되는 전체 파일 삭제 및 자동 리팩토링.

**리팩토링과 기능 추가 작업은 분리하여 제출하세요.** 기존 코드를 리팩토링하고 새로운 동작을 추가하는 변경 사항은 두 개의 변경 사항으로 간주하여 별도로 제출해야 합니다. 사소한 코드 정리(변수 이름 변경 등)는 검토자의 재량에 따라 포함될 수 있습니다.

## 설명 변경

모든 변경 사항에는 버전 관리 기록에서 독립적으로 존재하는 설명이 필요합니다.

**첫 번째 줄:** 간결하고, 명령형이며, 독립적이어야 합니다. "FizzBuzz RPC 삭제"이지 "FizzBuzz RPC 삭제 중"이 아닙니다. 변경 내역을 검색하는 사람이 차이점을 읽지 않고도 변경 내용을 이해할 수 있도록 충분히 설명적이어야 합니다.

**본문:** 무엇이 변경되었고 그 이유는 무엇입니까? 코드 자체에는 드러나지 않는 맥락, 결정 사항 및 추론 과정을 포함하십시오. 관련이 있는 경우 버그 번호, 벤치마크 결과 또는 설계 문서를 참조하십시오. 접근 방식의 한계가 있는 경우 이를 인정하십시오.

**반패턴:** "버그 수정", "빌드 수정", "패치 추가", "코드를 A에서 B로 이동", "1단계", "편의 함수 추가"

## 검토 프로세스

### 1단계: 상황 파악

코드를 살펴보기 전에 의도를 파악하세요.

```
- 이번 변화는 무엇을 달성하려는 것입니까?
- 어떤 사양이나 작업을 구현하는가?
- 예상되는 행동 변화는 무엇입니까?
```

### 2단계: 먼저 테스트 결과를 검토하세요

테스트를 통해 의도와 적용 범위를 파악할 수 있습니다.

```
- 해당 변경 사항에 대한 테스트가 있습니까?
- 구현 세부 사항이 아닌 동작 자체를 테스트하는 건가요?
- 예외적인 상황도 포함되나요?
- 검사에는 설명적인 이름이 있나요?
- 코드가 변경될 경우, 테스트가 회귀 오류를 잡아낼 수 있을까요?
```

### 3단계: 구현 검토

다음 다섯 가지 축을 염두에 두고 코드를 살펴보세요.

```
변경된 파일마다:
1. 정확성: 이 코드는 테스트에서 요구하는 대로 작동하는가?
2. 가독성: 도움 없이 이해할 수 있는가?
3. 아키텍처: 시스템에 적합한가?
4. 보안: 취약점이 있습니까?
5. 성능: 병목 현상이 있습니까?
```

### 4단계: 결과 분류

작성자가 필수 사항과 선택 사항을 알 수 있도록 모든 댓글에 심각도 등급을 표시하세요.

| 접두사 | 의미 | 저자 행위 |
|--------|---------|---------------|
| (접두사 없음) | 필수 변경 사항 | 병합 전 반드시 처리해야 함 |
| **심각:** | 블록 병합 | 보안 취약점, 데이터 손실, 기능 오류 |
| **사소한 지적:** | 사소하고 선택적인 사항 | 작성자는 서식, 스타일 설정 등을 무시해도 됩니다. |
| **선택 사항:** / **고려 사항:** | 제안 | 고려해 볼 만하지만 필수 사항은 아님 |
| **참고** | 정보 제공용 | 조치 필요 없음 - 향후 참조를 위한 맥락 정보입니다 |

이는 저자들이 모든 피드백을 필수적인 것으로 간주하고 선택적인 제안에 시간을 낭비하는 것을 방지합니다.

### 5단계: 검증을 확인합니다

저자의 검증 내용을 확인하세요:

```
- 어떤 테스트가 진행되었습니까?
- 빌드가 성공했나요?
- 변경 사항을 수동으로 테스트했습니까?
- 사용자 인터페이스 변경 사항에 대한 스크린샷이 있나요?
- 전후 비교 자료가 있나요?
```

### 현재 프로젝트 검증 기준

이 저장소(`bookbackend`)에서 코드 변경을 리뷰할 때는 다음 프로젝트 문서를 기준으로 검증 범위를 확인하십시오.

- `CLAUDE.md` — 명령어, 인증 방식, 공통 응답 형식
- `docs/WORKFLOW.md` — 개발/테스트 표준 흐름
- `docs/spec.md` 및 `docs/*-api.md` — API 명세
- `schema.prisma` — 데이터 모델과 마이그레이션 영향
- `test/testplan-*.md` 및 `test/test-*.ts` — 테스트 계획과 기존 테스트

리뷰 대상이 런타임 코드, Prisma 스키마, API 동작, 테스트 코드에 영향을 준다면 다음 검증 여부를 확인하십시오.

```bash
npx prisma validate
npm run build
```

Prisma 스키마나 마이그레이션이 변경되었으면 테스트 DB 기준으로 다음도 확인하십시오.

```bash
npx prisma migrate deploy
```

변경 범위가 특정 도메인에 한정되면 관련 도메인 테스트를 확인하십시오.

```bash
npm run test:auth
npm run test:book
npm run test:reading-log
npm run test:memo
npm run test:stats
```

공통 모듈, 라우팅, 세션, Prisma 설정, 테스트 헬퍼, 응답 형식에 영향이 있으면 전체 테스트를 확인하십시오.

```bash
npm test
```

테스트는 실제 MySQL 테스트 DB를 사용하므로 `.env.test`, `NODE_ENV=test`, 테스트 DB 분리, `cleanDb()` 보호 장치가 유지되는지도 확인하십시오.

## 멀티 모델 검토 패턴

다양한 검토 관점에 따라 서로 다른 모델을 사용하십시오.

```
모델 A가 코드를 작성합니다.
    │
    ▼
Model B의 정확성 및 아키텍처 검토
    │
    ▼
모델 A는 피드백을 다룹니다.
    │
    ▼
인간이 최종 결정을 내린다
```

이는 단일 모델이 놓칠 수 있는 문제를 포착합니다. 모델마다 사각지대가 다르기 때문입니다.

**리뷰 담당자에게 보내는 메시지 예시:**
```
이 코드 변경 사항의 정확성, 보안성 및 규정 준수 여부를 검토하십시오.
우리 프로젝트의 규칙입니다. 사양서에는 [X]라고 되어 있습니다. 변경 사항은 [Y]여야 합니다.
모든 문제에 대해 '심각', '중요' 또는 '제안'으로 표시해 주세요.
```

## 데드 코드 Hygiene

리팩토링이나 구현 변경 후에는 사용되지 않는 코드가 있는지 확인하십시오.

1. 현재 접근할 수 없거나 사용되지 않는 코드를 식별합니다.
2. 명시적으로 나열하십시오
3. **삭제하기 전에 반드시 물어보세요:** "이제 더 이상 사용하지 않는 요소들을 삭제해도 될까요?: [목록]?"

## 검토 속도

리뷰 진행이 느리면 팀 전체의 업무가 마비됩니다. 리뷰 작업을 위해 다른 업무로 전환하는 데 드는 비용이 다른 사람들이 기다려야 하는 시간보다 훨씬 적습니다.

- **영업일 기준 하루 이내 답변** — 이는 최대 소요 시간이며, 목표 소요 시간은 아닙니다.
- **이상적인 주기:** 집중 코딩 중이 아니라면 검토 요청이 도착한 직후에 응답하세요. 일반적인 변경 사항은 하루에 여러 번의 검토를 거쳐야 합니다.
- 최종 승인보다 **개별 응답을 신속하게 처리하는 것을 우선시하세요.** 여러 차례의 수정이 필요하더라도 빠른 피드백은 불만을 줄여줍니다.
- **대규모 변경 사항:** 작성자에게 하나의 큰 변경 사항 묶음을 검토하는 대신 변경 사항들을 나누어 검토해 달라고 요청하세요.

## 의견 불일치 처리

리뷰 관련 분쟁을 해결할 때는 다음 우선순위를 적용하세요.

1. **기술적 사실과 데이터**는 의견과 선호도보다 우선합니다.
2. **스타일 가이드**는 스타일 문제에 있어서 절대적인 권위를 지닌다.
3. **소프트웨어 설계**는 개인적인 선호가 아닌 공학적 원칙에 따라 평가되어야 합니다.
4. **코드베이스 일관성**은 전반적인 건전성을 저하시키지 않는다면 허용 가능합니다.

"나중에 정리할게요."라는 말은 받아들이지 마세요. 경험상 미뤄진 정리 작업은 거의 이루어지지 않습니다. 정말 긴급한 상황이 아니라면 제출 전에 정리를 완료하도록 요구하세요. 이번 변경으로 주변 문제를 해결할 수 없다면, 스스로 버그를 보고하고 담당자를 지정하도록 하세요.

## 솔직한 리뷰

본인이 작성했든, 다른 담당자가 작성했든, 사람이 작성했든, 코드를 검토할 때는 다음 사항을 고려하세요.

- **무턱대고 승인하지 마세요.** 검토 증거 없이 "LGTM"이라고만 하는 것은 누구에게도 도움이 되지 않습니다.
- **실제 문제를 완화하지 마세요.** 실제 운영 환경에 영향을 미칠 버그에 대해 "이건 사소한 문제일 수도 있습니다"라고 말하는 것은 정직하지 못한 처사입니다.
- **가능하면 문제를 수치화하세요.** "이 N+1 쿼리는 목록의 항목당 약 50ms를 추가합니다."라고 하는 것이 "느릴 수 있습니다."라고 하는 것보다 더 좋습니다.
- **명백한 문제가 있는 접근 방식에 대해서는 이의를 제기하십시오.** 아첨은 검토 과정에서 실패의 원인입니다. 구현에 문제가 있다면 직접적으로 지적하고 대안을 제시하십시오.
- **변경 요청을 정중하게 수용하세요.** 작성자가 전체적인 맥락을 알고 있고 동의하지 않는다면, 그들의 판단을 존중하세요. 코드에 대한 의견을 제시하고, 사람을 비난하지 마세요. 개인적인 비판은 코드 자체에 초점을 맞추도록 바꿔보세요.

## 의존성 규율

코드 리뷰의 일부로 의존성 리뷰도 포함됩니다.

**종속성을 추가하기 전에:**
1. 기존 스택이 이 문제를 해결합니까? (대부분 그렇습니다.)
2. 의존성의 규모는 어느 정도입니까? (번들 영향도를 확인하세요.)
3. 활발하게 유지보수되고 있습니까? (최근 커밋, 미해결 이슈 확인)
4. 알려진 취약점이 있습니까? (`npm audit`)
5. 라이선스는 무엇인가요? (프로젝트와 호환되어야 합니다.)

**규칙:** 새로운 종속성보다는 표준 라이브러리와 기존 유틸리티를 우선적으로 사용하세요. 모든 종속성은 부담입니다.

### 판결
- [ ] **승인** — 병합 준비 완료

## 위험 신호

- PR이 검토 없이 병합되었습니다
- 테스트 통과 여부만 확인하는 검토 (다른 요소는 무시)
- 실제 검토 증거 없이 "LGTM"이라고 표시함
- 보안에 민감한 변경 사항을 보안 검토 없이 진행
- "제대로 검토하기에는 너무 큰" 대규모 PR은 분할하세요.
- 버그 수정 PR에는 회귀 테스트가 포함되지 않습니다.
- 중요도 표시가 없는 리뷰 댓글은 필수 항목과 선택 항목을 구분하기 어렵게 만듭니다.
- "나중에 고칠게요"라는 말을 받아들이는 것 — 그런 일은 절대 일어나지 않아요
